TO JEST NOWY PROCES PRACY
Nie zaczynasz od wielkiej przebudowy pluginu i nie mieszasz UX z runtime.
Pracujesz etapami:

ETAP 1 - UI PROTOTYPE FIRST
- najpierw projektujesz i budujesz zatwierdzalny prototype UI/UX
- bez integracji z WP runtime
- bez migracji
- bez dotykania bootstrapa pluginu
- celem jest zaakceptowany model interfejsu

ETAP 2 - STABLE PLUGIN BASELINE
- dopiero po akceptacji prototype UI budujesz nowy plugin
- najpierw stabilna instalacja i aktywacja
- potem auth + shell + launcher
- potem moduły
- Panel Ofertowy legacy integrowany ostrożnie, bez regresji

ETAP 3 - RUNTIME QA
- dopiero po stabilnej aktywacji i poprawnym UI przechodzisz do testów runtime

NAJWAŻNIEJSZE LEKCJE Z PIERWSZEJ PRÓBY
1. Nie wolno mieszać zmian UX z bootstrapem pluginu w tej samej iteracji.
2. Nie wolno projektować fullscreen admin panelu. Kierunek to kompaktowe okno aplikacji premium.
3. Nie wolno robić kolejnych "retuszy CSS" bez zmiany modelu UX.
4. Nie wolno duplikować plików, robić bałaganu w repo, mnożyć preview, zipów i eksperymentalnych wersji bez kontroli.
5. Nie wolno wrzucać paczek, które nie dają się aktywować.
6. Nie wolno zostawiać BOM w PHP ani rozwalać UTF-8.
7. Clean start ma być naprawdę czysty i nie może automatycznie migrować legacy danych przy aktywacji.
8. Mobile nie może być ściśniętym desktopem.
9. Panel Ofertowy legacy to referencja jakości UI/UX i nie może być degradowany.
10. Najpierw ma być zaakceptowany model interfejsu, potem dopiero wdrożenie do pluginu.
